📖 Context Note

This is a MERN educational curriculum (TLM / The Last Mile) — the movie-buzz-finished directory serves as the instructor's reference implementation. The evaluation is intentionally held to production standards to surface every teachable gap. The week_16 version shows clear architectural improvements (service layer, error state UI) but its CRUD functions are all marked TODO — so the movie-buzz-finished server is the live baseline used throughout this audit.

What's Working Well

The project demonstrates solid pedagogical progression: React Router v6 integration is functional, the component decomposition (Header/Footer, MoviesList, MovieForm, SearchBar, FilterControls) follows good separation of UI concerns, the Mongoose schema uses timestamps, a seed script is included, and client-side filtering + sorting logic is correctly implemented. The Tic-Tac-Toe checkpoint server (week_10) is actually the most production-ready server in the repo — it has env-based PORT, a proper 404 handler, a global error middleware, and input validation.

The Core Problem

Think of the server like a restaurant kitchen with no locks on the back door, no allergy checks, and no ticket system. Guests can order anything, staff serve it blind, and if the fryer catches fire nobody has a plan. The five critical findings below would stop a real deployment dead on day one:

CORS wildcard Hardcoded secrets Mass assignment Hardcoded client URLs No .env usage

02 Architecture Evaluation

Project Structure

The directory layout is well-organized with a clean client/server split. The weekly progression from static HTML → React state → Express → MongoDB → CRUD is a textbook MERN teaching arc. That said, the finished product still has architectural patterns that belong in week 3, not week 16.

⚙️ Monolithic index.jsx

All application state (movies, loading, search/filter state), all API calls (axios.get/post/put/delete), all filtering logic, and all routing live in a single root file. This is the React equivalent of putting your entire game save, inventory system, and world map in one God object. The week_16 version correctly introduces a movieService.js layer — that pattern needs to be back-ported to movie-buzz-finished.

⚙️ No Proxy / Environment Config

The client and server communicate via a hardcoded absolute URL (http://localhost:4000). CRA's package.json "proxy" field exists for exactly this — set it once and all /api/* calls route automatically. This is also a prerequisite for any real deployment (Vercel, Heroku, Railway, etc.).

🔧 No Error Boundary

React's component tree has no ErrorBoundary wrapper. A single runtime crash (like movie.genre.join() on bad data) will unmount the entire UI silently. Like a single rogue ingredient crashing the whole menu — one wrapper component contains the blast radius.

⚙️ Dead Controller Code: getMovieToEdit

server/controllers/movieController.js defines a getMovieToEdit function. server/routes/movieRoutes.js has no route that calls it. The CLAUDE.md even lists GET /api/movie/:id in the API docs, but it doesn't exist. This is dead code and a documentation lie — both are technical debt.

Data Flow Diagram (Current vs. Recommended)

Currently: index.jsx → raw axios → Express → Mongoose → MongoDB

Recommended: index.jsx → movieService.js → Express (with validation middleware) → Mongoose (with schema validators) → MongoDB

03 Security Audit

🔴 CRITICAL — CORS Wildcard (app.use(cors()))

server/server.js line 10 — No origin restriction. Any website in the world can make authenticated cross-origin requests to your API. In production with a real database, this enables cross-site request forgery from arbitrary origins. Compare to the week_10 TTT server which correctly scopes to origin: 'http://localhost:3000'.

❌ BEFORE
// Accepts requests from literally anywhere on the internet
app.use(cors())
✅ AFTER
const allowedOrigins = [
  'http://localhost:3000',
  process.env.CLIENT_URL // e.g. 'https://yourapp.vercel.app'
].filter(Boolean);

app.use(cors({
  origin: (origin, callback) => {
    if (!origin || allowedOrigins.includes(origin)) callback(null, true);
    else callback(new Error('Not allowed by CORS'));
  },
  credentials: true
}));
🔴 CRITICAL — Hardcoded MongoDB Connection String

server/config/db_connection.js line 2 — The connection string is baked directly into source code. If this project ever connects to a real MongoDB Atlas cluster with credentials, those credentials are now in git history forever. This is how data breaches start. Use .env + dotenv.

❌ BEFORE
const connectionString = "mongodb://127.0.0.1/movie-buzz"
✅ AFTER
// .env file (never commit this):
// MONGO_URI=mongodb://127.0.0.1/movie-buzz
// PORT=4000

// db_connection.js:
require('dotenv').config();
const connectionString = process.env.MONGO_URI;
if (!connectionString) throw new Error('MONGO_URI not set in .env');
🔴 CRITICAL — Mass Assignment Vulnerability

server/controllers/movieController.js — Movie.create(body) and Movie.findByIdAndUpdate(id, body, ...) pass the raw request body directly to Mongoose. An attacker can inject any field they want — including Mongoose operators like $where, extra schema fields, or attempts to overwrite _id. This is the NoSQL equivalent of SQL injection. Always whitelist fields explicitly.

❌ BEFORE
createMovie(req, res) {
  const { body } = req
  Movie.create(body)  // ← entire req.body goes straight to DB
}
✅ AFTER
createMovie(req, res) {
  const { name, description, rating, length, year, genre, director, stars } = req.body
  Movie.create({ name, description, rating, length, year, genre, director, stars })
    .then(movie => res.status(201).json(movie))
    .catch(err => res.status(500).json({ error: err.message }))
}
🔴 CRITICAL — Hardcoded API Base URL in Client

client/src/index.jsx — http://localhost:4000 is hardcoded in every axios call. This application will not work the moment it's deployed anywhere that isn't your local machine. Fix: add "proxy": "http://localhost:4000" to client/package.json and use relative paths /api/movies in dev; use REACT_APP_API_URL env var for production.

🔴 CRITICAL — PORT Hardcoded (No Process Env Fallback)

server/server.js line 14 — const PORT = 4000. Every PaaS (Heroku, Railway, Render) assigns ports dynamically via process.env.PORT. Hardcoding means the server will fail to start on any cloud environment. Fix: const PORT = process.env.PORT || 4000. The week_10 TTT server already does this correctly.

04 Code Quality & Resilience

🟠 HIGH — Race Condition in MovieForm Submit

client/src/components/MovieForm/MovieForm.jsx — editMovie(movieData) and addMovie(movieData) are called synchronously, then finish() immediately navigates away. The axios calls are async — by the time the server responds, the component is already gone. This means a failed API call is completely silently ignored. The user sees "success" even on a 500 error. Furthermore, window.location = "/" causes a full page reload (SPA suicide — defeats the entire point of React Router).

❌ BEFORE
const finish = () => {
  setLoading(false);
  window.location = "/";  // 💀 full page reload
};
if (movies) {
  editMovie(movieData);  // fires async, not awaited
  finish();             // navigates before server responds
}
✅ AFTER
import { useNavigate } from 'react-router-dom';
const navigate = useNavigate();

const handleSubmit = async (event) => {
  event.preventDefault();
  const validation = validate();
  if (Object.keys(validation).length > 0) { setErrors(validation); return; }
  setLoading(true);
  try {
    if (movies) await editMovie(movieData);
    else        await addMovie(movieData);
    navigate('/');  // React Router: no page reload
  } catch (err) {
    setErrors({ submit: err.message || 'Something went wrong. Please try again.' });
  } finally {
    setLoading(false);
  }
};
🟠 HIGH — MovieBlock.jsx Runtime Crash on Bad Data

client/src/components/Utility/MovieBlock.jsx — movie.genre.join(', ') and movie.stars.join(', ') are called unconditionally. If genre or stars is undefined, null, or accidentally stored as a string (which the form currently handles via comma-split, but older DB records may not), this throws TypeError: movie.genre.join is not a function and crashes the entire component tree. Zero guard = zero resilience.

❌ BEFORE
<p><strong>Genre:</strong> {movie.genre.join(', ')}</p>
✅ AFTER
<p><strong>Genre:</strong> {
  Array.isArray(movie.genre) ? movie.genre.join(', ') : (movie.genre || 'N/A')
}</p>
🟠 HIGH — Direct Link to /edit/:id Renders Empty Form

client/src/components/MovieForm/MovieForm.jsx — The edit form pre-populates by filtering the movies prop array client-side. If the user navigates directly to /edit/abc123 (bookmarked, shared, or refreshed), the movies prop will be empty until the fetch completes — meaning the form renders blank and the useEffect that sets defaults runs before data arrives. The fix is to either: (a) fetch the specific movie by ID from the API on mount, or (b) wait for the parent loading state before rendering the form. The dead getMovieToEdit controller route was clearly built for (a) — it just never got wired up.

🟠 HIGH — No User-Facing Error State

client/src/index.jsx — Every .catch() block calls console.error and nothing else. If the MongoDB server goes down, the client shows a blank list with no explanation. If a delete fails, the user doesn't know. The week_16 App.js already has const [error, setError] = useState(null) with an ErrorMessage component — that pattern needs to be in the finished version.

🟠 HIGH — No Server-Side Input Validation

server/controllers/movieController.js — The server performs zero validation before writing to the database. A movie with no name, a year of -99999, or a rating of "YOLOstar" will all be accepted. Client-side validation (in MovieForm.jsx) is good UX but never a substitute for server-side checks — anyone using curl or Postman bypasses it entirely. Add validation middleware or use Mongoose's built-in schema validators.

✅ Add Mongoose validators to movie.js
const movieSchema = new Schema({
  name:        { type: String,  required: [true, 'Movie name is required'], trim: true, minlength: 2 },
  rating:      { type: String,  enum: ['G', 'PG', 'PG-13', 'R', 'NC-17', 'NR'] },
  year:        { type: Number,  min: 1888, max: new Date().getFullYear() + 2 },
  genre:       { type: [String], required: true },
  director:    { type: String,  trim: true },
  stars:       { type: [String] },
  description: { type: String,  trim: true },
  length:      { type: String,  trim: true }
}, { timestamps: true })
🟡 MEDIUM — exact Prop Deprecated in React Router v6

client/src/index.jsx — <Route exact path="/"> passes an unknown prop in RRv6 (exact matching is the default). This generates browser console warnings. Simple fix: remove all exact props.

🟡 MEDIUM — Hardcoded Genre List in FilterControls

client/src/components/SearchFilter/FilterControls.jsx — The 12 genre options are hardcoded. The moment a user adds a movie with genre "Western" (which exists in the test data!) or "Documentary", it won't appear in the filter. Derive the genre list dynamically from the movies prop using new Set(movies.flatMap(m => m.genre)).values() and pass it in.

05 Testing Audit

🟠 HIGH — Tests Run Against the Production Database

server/test/test_helper.js — The test helper connects to mongodb://127.0.0.1/movie-buzz — the same connection string as the running dev server. The beforeEach hook calls movies.drop(), which wipes all data before every test. If a developer runs npm test while the dev server is running, their entire movie collection gets nuked. A separate test DB (movie-buzz-test) gated by NODE_ENV=test is essential.

🟡 MEDIUM — No HTTP Status Assertions in Tests

server/test/movies_test.js — All 3 tests use .end() callbacks without checking the response status code. If the POST returns 500 and the Philadelphia Story never gets created, the test's Movie.findOne() call would return null, causing assert(null.length === "112 minutes") — which throws a TypeError rather than a clean assertion failure. Always .expect(201) or .expect(200) before the .end().

🟡 MEDIUM — Only 3 Tests, Missing GET + Validation Coverage

The test suite covers CREATE, UPDATE, and DELETE. Missing: (1) GET /api/movies — the most-called route in the app, (2) GET /api/movie/:id — which the CLAUDE.md docs but the routes file doesn't implement, (3) invalid inputs — what happens on a POST with a missing name? A PUT to a non-existent ID? A DELETE with a malformed ObjectId? Edge cases are where bugs live.

🟢 LOW — useFindAndModify: false Deprecated in Mongoose 6+

server/test/test_helper.js line 8 — This option was removed in Mongoose 6. It will generate a warning at minimum and can throw on newer patch versions. Remove it entirely — Mongoose 6+ no longer uses the deprecated findAndModify driver method.

🟢 LOW — mocha in dependencies, Not devDependencies

server/package.json — mocha and supertest are listed under dependencies. They should be in devDependencies — test runners don't belong in a production bundle. This inflates the install footprint in any CI/CD or containerized deployment.
